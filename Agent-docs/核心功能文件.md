# Jarvis Platform V1 —— Token 计费系统设计文档

## 一、设计目标

Jarvis Platform 未来可能会接入不同的大模型（Gemini、OpenAI、DeepSeek 等）、不同的 TTS（ElevenLabs 等）以及 STT 服务。

因此，**平台的计费不能直接依赖任何一家 API 的计费规则。**

平台需要拥有自己的虚拟货币：

> **Platform Tokens（平台余额）**

用户永远只看到 Platform Tokens，不关心底层调用了什么 AI 服务。

---

# 二、核心理念

## Platform Tokens ≠ LLM Tokens

Platform Tokens 只是平台货币。

例如：

```text
10000 Platform Tokens
```

它代表余额。

不是 Gemini Token。

不是 OpenAI Token。

不是 ElevenLabs Characters。

后台负责自动换算。

---

## 用户看到的世界

例如：

```
余额：

50000 Tokens
```

聊天：

```
本次消耗：

28 Tokens
```

结束。

用户无需知道：

- Gemini 消耗多少 token
- ElevenLabs 消耗多少字符
- 成本是多少

这些全部属于后台逻辑。

# 十九、V1 开发原则（非常重要）

所有 AI 服务都必须实现统一接口，而不是在业务代码里直接调用第三方 SDK。

例如：

```text
Runtime
    │
    ├── LLMProvider（Gemini、OpenAI、DeepSeek）
    ├── TTSProvider（ElevenLabs、Azure TTS）
    └── STTProvider（Google STT、Whisper）
           │
           ▼
      CostService（统一计算成本）
           │
           ▼
     BillingService（扣除余额、记录账单）
```

这样以后更换供应商时，只需要替换 Provider 层，Runtime、计费、前端都无需修改。

## 最后的建议（建议直接交给 Claude 实现）

这套计费系统不要写死任何价格。

建议增加一张**价格配置表**（或配置文件），例如：

```yaml
pricing:
  markup: 1.5
  platform:
    tokens_per_rmb: 1000

  gemini:
    input_per_million: 1.0
    output_per_million: 2.0

  elevenlabs:
    price_per_100k_characters: 35.0
```

`CostService` 每次都读取这份配置进行计算。

这样未来：

- Gemini 涨价
- ElevenLabs 降价
- 利润倍率从 1.5 调整到 1.8
- Platform Tokens 汇率调整

**都无需修改任何业务代码，只需要修改配置即可。**

我认为这是最符合企业级设计的实现方式，也非常适合作为 Jarvis Platform V1 的计费基础。
